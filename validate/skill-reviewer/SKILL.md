# Skill Reviewer — 结构化审查工具

## 核心理念

审查的价值在于**系统性**而非即兴评价。通过将审查标准固化为清单，确保每次审查的一致性和可追溯性。这个 skill 将"审查标准"和"审查逻辑"分离——标准放在 `references/` 下的清单文件中，逻辑写在这里。这样做的好处是：添加新的审查类型只需要增加一份清单文件，无需修改 skill 本身。

## 目录结构

```
skill-reviewer/
├── SKILL.md                          # 本文件：审查逻辑与工作流
└── references/
    ├── python-review-checklist.md     # Python 代码审查清单
    ├── owasp-checklist.md             # OWASP Top 10 安全检查清单
    └── doc-quality-checklist.md       # 文档质量检查清单
```

需要新的审查类型时，在 `references/` 下创建对应的 `xxx-checklist.md` 文件即可。

## 工作流程

### 第一步：识别审查类型

根据用户的请求和待审查内容，确定应该加载哪份审查清单：

| 用户意图 | 加载清单 |
|---------|---------|
| Python 代码审查、代码风格、代码质量 | `references/python-review-checklist.md` |
| 安全审计、漏洞检查、安全评估 | `references/owasp-checklist.md` |
| 文档审查、文档质量、技术文档检查 | `references/doc-quality-checklist.md` |

如果用户没有明确指定类型，根据待审查内容自动判断：
- `.py` 文件 → Python 代码审查
- 涉及用户输入、认证、数据库的代码 → 安全审计
- `.md` / `.rst` / 纯文本文档 → 文档质量检查
- 如果内容跨越多个类型，可以组合加载多份清单

如果 `references/` 下没有匹配的清单，告知用户当前支持的审查类型，并建议用户创建新清单或从通用审查角度给出建议。

### 第二步：加载审查清单

读取对应的 `references/xxx-checklist.md` 文件。清单中的每个条目都有唯一编号，审查结果需要引用这些编号以便追溯。

### 第三步：逐条检查

对照清单中的每个条目，检查待审查内容。注意：
- 只报告有实际问题的条目，不要列出所有通过的条目（那样会让报告臃肿且无用）
- 每个发现必须有清单条目编号作为依据——不要凭空发明问题
- 如果发现了清单中没有覆盖但确实重要的问题，可以作为"清单外发现"单独列出

### 第四步：输出审查报告

按照以下格式输出结构化审查报告：

```markdown
# 审查报告

**审查类型**: [Python 代码审查 / 安全审计 / 文档质量检查]
**审查清单**: [所用清单文件名]
**审查对象**: [文件名或描述]

## 总览

共发现 X 个问题：严重 X 个，警告 X 个，建议 X 个。

## 严重问题（必须修复）

### [清单编号] 问题标题
- **位置**: 文件名:行号 或具体段落
- **问题**: 问题的具体描述
- **修复建议**: 具体的修复方案，最好附上修改后的代码/文本

## 警告问题（建议修复）

### [清单编号] 问题标题
- **位置**: ...
- **问题**: ...
- **修复建议**: ...

## 优化建议（可选改进）

### [清单编号] 问题标题
- **位置**: ...
- **问题**: ...
- **修复建议**: ...

## 清单外发现（如有）

### 问题标题
- **位置**: ...
- **问题**: ...
- **修复建议**: ...
- **建议添加到清单**: 是/否
```

### 严重程度判断标准

- **严重**：会导致程序崩溃、数据丢失、安全漏洞、或严重误导读者的问题
- **警告**：不会立即出错但可能引发后续问题，或违反团队约定的规范
- **建议**：代码/文档可以工作但有改进空间，属于最佳实践层面

## 使用示例

**示例 1：用户请求审查 Python 代码**

用户说："帮我看看这个 Python 文件有没有问题"

→ 加载 `references/python-review-checklist.md`，逐条检查代码风格、类型标注、异常处理、命名规范等。

**示例 2：用户请求安全审计**

用户说："这个 API 接口有安全隐患吗"

→ 加载 `references/owasp-checklist.md`，重点检查注入攻击、认证授权、敏感数据暴露等。

**示例 3：组合审查**

用户说："全面检查一下这个项目"

→ 根据文件类型组合加载多份清单，分别输出各类审查结果。
