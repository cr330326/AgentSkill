# Inversion Skill — 先问再做

## 为什么需要反转

Agent 的默认倾向是接到任务立刻开始执行。对简单任务这没问题，但对复杂任务（项目规划、
架构设计、迁移方案等），贸然执行几乎必然导致：

1. **方向错误** — 猜错了用户的核心需求，整个产出需要推倒重来
2. **遗漏关键约束** — 忽略了技术栈限制、合规要求、性能指标等硬约束
3. **过度工程** — 没搞清楚范围就开始做，做了大量不需要的东西

反转模式的核心思想：**花 2 分钟提问，省 20 分钟返工。**

## 何时触发

识别到以下信号时启用反转模式：

- 用户请求涉及**多个可能的执行方向**（"帮我做个后台管理系统" — 用什么框架？多少个模块？）
- 请求中**缺少关键信息**（"帮我设计API" — 什么业务？给谁用？什么协议？）
- 请求属于**高返工成本类型**（架构设计、数据库迁移、基础设施搭建）
- 用户明确说了"规划"、"设计"、"评估"等需要前期调研的词

### 不要触发

- 用户说"修复这个bug"且已贴出错误信息和代码 → 直接修
- "帮我加个注释" → 直接做
- "XXX是什么意思？" → 直接答
- 用户在第一条消息中已经给了详尽的上下文（技术栈、约束、期望输出格式）→ 直接执行

## 访谈流程

### 第一步：识别场景类型

根据用户请求判断属于哪个场景类型，然后加载对应的问题模板：

| 场景 | 参考文件 | 核心关注点 |
|------|---------|-----------|
| 项目规划 | `references/questions-project.md` | 范围、技术栈、时间线、团队 |
| 需求确认 | `references/questions-requirements.md` | 用户故事、优先级、验收标准 |
| 复杂任务拆解 | `references/questions-decomposition.md` | 依赖关系、里程碑、风险 |
| 配置初始化 | `references/questions-config.md` | 环境、参数、兼容性 |
| 技术方案选型 | `references/questions-tech-selection.md` | 候选方案、评估维度、约束 |
| 接口/数据模型设计 | `references/questions-api-design.md` | 消费者、协议、数据流 |
| 迁移/重构规划 | `references/questions-migration.md` | 现状、目标态、兼容性、回滚 |
| 故障排查 | `references/questions-debugging.md` | 症状、环境、时间线、已尝试方案 |

如果请求不完全落入单一场景，组合使用多个模板中的相关问题。

### 第二步：动态生成问题

模板是**起点而非终点**。根据以下原则动态生成和调整问题：

**好问题的标准：**
- **具体** — "你期望的 API 响应时间上限是多少？" 而不是 "你对性能有什么要求？"
- **封闭优先** — 能用选择题就不用开放题（"用 REST 还是 GraphQL？" 优于 "你想用什么协议？"）
- **有默认值** — "数据库用 PostgreSQL 还是 MySQL？默认推荐 PostgreSQL" 让用户可以快速确认
- **渐进深入** — 先问大方向，再问细节；避免一上来就问极细的配置项

**问题生成策略：**
1. 先从模板中选出与当前请求最相关的 3-5 个问题
2. 根据用户的回答动态追问 — 如果用户说"用 React"，追问"状态管理用什么？有现有代码库吗？"
3. 发现新的未知信息时及时补充新问题
4. 不要机械地念模板，跳过用户已经回答过的问题

### 第三步：提问方式

**每次提问规则：**
- 一次提 1-3 个问题，不要一次性倒出所有问题（信息轰炸会让用户反感）
- 对每个问题简短解释为什么需要这个信息（1 句话）
- 用 `question` 工具（如果可用）提供选项让用户快速选择
- 如果没有 `question` 工具，用编号列表并标注推荐选项

**提问示例：**

```
在开始设计之前，我需要确认几个关键信息：

1. **目标用户是谁？** 内部团队用还是对外开放？
   （这决定了认证方案和 API 风格的选择）

2. **预期并发量级？** 如：< 100 QPS / 100-1000 QPS / > 1000 QPS
   （这影响架构复杂度和基础设施选型）

3. **是否需要与现有系统集成？** 如果是，哪些系统？用什么协议？
   （这决定了接口设计的约束条件）
```

### 第四步：判断何时停止提问

**停止条件（满足任一即可）：**

1. **信息充分** — 你有足够的信息来开始执行，且对执行方向有高度置信
2. **收益递减** — 继续问下去获得的新信息对执行方向的影响已经很小
3. **用户信号** — 用户说了"够了"、"开始吧"、"差不多了"、"你先做着看"等
4. **上下文饱和** — 核心约束（范围、技术栈、时间线、硬性要求）都已明确

**不停止的信号：**
- 你还不确定用哪个技术方案（有 2+ 个同样合理的选择且用户没表态）
- 关键约束仍然未知（比如"数据库可以随便选"但你不知道有没有现有数据库要兼容）
- 用户的回答引出了新的重要分支（"哦对了我们还需要支持多租户"）

**经验法则：** 大多数场景在 5-8 个问题内可以收集到足够信息。如果超过 10 个问题
还没收集完，考虑是否在问不重要的细节 — 这些可以在执行过程中边做边确认。

### 第五步：总结确认后执行

提问完成后，在开始执行前做一次**信息摘要确认**：

```
基于你的回答，我确认以下方案：

- **项目类型**：内部管理后台
- **技术栈**：React + TypeScript + Ant Design
- **后端**：Node.js + Express + PostgreSQL
- **部署**：Docker Compose，部署到内部服务器
- **核心功能**：用户管理、权限控制、数据看板（共 3 个模块）
- **时间线**：2 周内完成 MVP

如果以上理解正确，我现在开始执行。有需要调整的地方请告诉我。
```

等用户确认后再开始执行。如果用户对摘要中的某项提出修改，更新后再确认一次。

## 执行转换

从访谈模式切换到执行模式后：

1. 将收集到的所有信息整理为**执行清单**（用 TodoWrite 工具）
2. 按优先级排序执行
3. 执行过程中如果发现遗漏的信息点，直接在当前步骤中追问，不需要重新进入完整的访谈流程

## 常见陷阱

| 陷阱 | 后果 | 避免方法 |
|------|------|---------|
| 问太多问题 | 用户不耐烦，体验差 | 控制在 5-8 个问题内；优先问影响执行方向的大问题 |
| 问太少就开始做 | 做出来的东西和用户要的不一样 | 确保核心约束（范围、技术栈、时间线）至少被覆盖 |
| 一次性问所有问题 | 信息轰炸 | 每次 1-3 个，渐进式 |
| 不解释为什么问 | 用户觉得在被审问 | 每个问题附 1 句解释 |
| 机械念模板 | 问了不相关的问题 | 动态裁剪，跳过已知信息 |
| 忘了总结确认 | 理解偏差到执行才发现 | 执行前必须输出信息摘要 |
