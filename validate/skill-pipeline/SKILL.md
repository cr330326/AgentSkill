# Pipeline — 流水线模式 Skill

## 这个 Skill 是什么

Pipeline 是一种**执行控制模式**。它解决的核心问题是：

> 复杂任务如果让 AI 一口气跑完，中间某步出错会导致后续全部白做，
> 回退成本极高。流水线模式在关键节点强制暂停，等人类确认后再继续，
> 把"一次性赌博"变成"分段可控"。

这个 skill 提供两层能力：
1. **内置流水线**：`references/` 下有现成的流水线定义，直接加载即可执行
2. **流水线模板**：按照本文档的格式，可以为任何场景创建自定义流水线

## 内置流水线

| 流水线 | 文件 | 适用场景 |
|--------|------|----------|
| 文档生成 | `references/doc-pipeline.md` | 技术文档、API 文档、用户手册、报告 |
| 版本发布 | `references/release-pipeline.md` | npm/PyPI/Docker 等包的版本发布 |
| 数据处理 | `references/data-pipeline.md` | ETL、数据清洗、批量转换、迁移 |

用户提到相关场景时，加载对应的流水线定义文件。

## 执行规则

以下规则是流水线模式的核心约束，无论内置还是自建流水线都必须遵守：

1. **PL-01 严格顺序**：按流水线定义的阶段顺序执行，禁止跳过任何阶段
2. **PL-02 检查点暂停**：标记为 `[CHECKPOINT]` 的阶段完成后，必须输出阶段结果摘要，然后停下来等待用户确认
3. **PL-03 确认才继续**：用户明确说"继续"、"下一步"、"OK"、"通过"等肯定词后，才能进入下一阶段；用户如果说"修改"、"重做"、"不对"，则在当前阶段重新执行
4. **PL-04 进度可视**：每个阶段开始时输出进度指示，格式为 `[阶段 N/总数] 阶段名称`
5. **PL-05 失败回退**：如果某阶段执行失败，停在当前阶段报告错误，不要尝试跳到下一阶段
6. **PL-06 最终汇报**：所有阶段完成后，输出完整的执行摘要，列出每个阶段的结果

## 流水线定义格式

每条流水线定义文件遵循以下结构：

```markdown
# <流水线名称>

## 适用场景
描述什么时候应该使用这条流水线。

## 阶段定义

### 阶段 1: <名称> [CHECKPOINT]
- **目标**: 这个阶段要完成什么
- **输入**: 需要什么信息/文件
- **操作**: 具体执行步骤
- **输出**: 阶段产物
- **检查要点**: 用户确认时应关注什么

### 阶段 2: <名称>
- **目标**: ...
- **操作**: ...
- **输出**: ...

（没有 [CHECKPOINT] 标记的阶段会自动执行，不暂停）

## 注意事项
特殊情况处理、常见陷阱等。
```

关键设计决策：
- **不是每个阶段都需要检查点**：只在"做错了代价很高"的节点标 `[CHECKPOINT]`。太多检查点会让用户疲于确认，太少则失去流水线的意义
- **阶段粒度**：一个阶段应该是 1-3 个相关操作的集合，不要把单个命令拆成一个阶段，也不要把 10 件不相关的事塞进一个阶段
- **输入/输出链**：上一阶段的输出应该是下一阶段的输入，形成清晰的数据流

## 如何创建自定义流水线

如果内置流水线不满足需求，可以参照上面的格式创建新的流水线定义。

### 步骤

1. **确定阶段划分**：把任务拆成 3-8 个阶段（少于 3 个阶段不需要流水线，超过 8 个考虑合并）
2. **标记检查点**：在以下类型的阶段标记 `[CHECKPOINT]`：
   - 不可逆操作之前（如发布、删除、发送）
   - 需要人类判断质量的节点（如文档审核、设计确认）
   - 涉及外部系统交互之前（如 API 调用、数据库写入）
3. **定义输入输出**：确保每个阶段的输入/输出链是连贯的
4. **写入 references/**：保存为 `references/<name>-pipeline.md`

### 示例：API 对接流水线

```markdown
# API 对接流水线

## 适用场景
对接第三方 API，从调研到上线的全流程。

## 阶段定义

### 阶段 1: API 调研 [CHECKPOINT]
- **目标**: 理解 API 文档，确认可行性
- **操作**: 阅读文档、列出所需端点、确认认证方式、记录速率限制
- **输出**: API 调研摘要（端点列表、认证方式、限制条件）
- **检查要点**: 端点是否覆盖需求？认证方式是否可接受？

### 阶段 2: 编写适配层
- **目标**: 实现 API 客户端封装
- **操作**: 创建客户端类、实现认证、封装请求/响应
- **输出**: API 客户端代码

### 阶段 3: 集成测试 [CHECKPOINT]
- **目标**: 验证 API 调用正确性
- **操作**: 编写测试用例、用沙盒环境测试、验证错误处理
- **输出**: 测试报告（通过/失败/覆盖率）
- **检查要点**: 测试是否覆盖主要场景？错误处理是否合理？

### 阶段 4: 上线部署
- **目标**: 部署到生产环境
- **操作**: 配置生产密钥、部署、监控首批请求
- **输出**: 部署确认 + 监控截图
```

## 检查点交互格式

到达检查点时，按以下格式输出：

```
---
[阶段 2/5] 文档结构审核 ✅ 完成

**阶段产物：**
- 生成了 5 个章节的大纲
- 每章包含 3-5 个小节

**需要您确认：**
1. 章节划分是否合理？
2. 是否需要增加/删除某些章节？

请回复"继续"进入下一阶段，或告诉我需要修改的地方。
---
```

## 常见问题

**Q: 用户说"不用每次都问我，直接做完"怎么办？**
尊重用户意愿。可以跳过中间检查点，但保留最后一个（通常是不可逆操作前的那个）。告知用户"我会在最终操作前再确认一次"。

**Q: 流水线执行到一半用户想改前面的步骤？**
回退到目标阶段重新执行。已经产生的后续阶段产物需要标记为失效，从目标阶段重新往后走。

**Q: 阶段之间需要用户提供额外输入怎么办？**
在阶段定义的"输入"字段标明需要用户提供什么。到达该阶段时主动询问。
