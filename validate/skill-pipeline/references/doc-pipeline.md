# 文档生成流水线

## 适用场景

需要生成结构化文档时使用：技术文档、API 文档、用户手册、项目报告、设计文档等。
适合文档篇幅较长（>5页）或需要多轮修改确认的场景。

## 阶段定义

### 阶段 1: 需求澄清 [CHECKPOINT]

- **目标**: 明确文档的范围、受众和格式要求
- **输入**: 用户的文档需求描述
- **操作**:
  1. 确认文档类型（技术文档 / 用户手册 / API 文档 / 报告）
  2. 确认目标读者（开发者 / 产品经理 / 最终用户 / 管理层）
  3. 确认输出格式（Markdown / Word / PDF / HTML）
  4. 确认篇幅预期和交付标准
  5. 如果用户提供了参考文档或模板，先阅读理解
- **输出**: 需求确认清单（文档类型、受众、格式、篇幅、特殊要求）
- **检查要点**: 需求理解是否准确？是否遗漏了重要约束？

### 阶段 2: 大纲设计 [CHECKPOINT]

- **目标**: 设计文档的章节结构
- **输入**: 阶段 1 的需求确认清单
- **操作**:
  1. 根据文档类型选择标准结构（见下方"常用文档结构"）
  2. 列出所有章节和小节标题
  3. 为每个小节标注预估篇幅（短/中/长）
  4. 标注哪些小节需要用户提供素材/数据
- **输出**: 完整的文档大纲（带层级编号）
- **检查要点**: 章节划分是否合理？是否有遗漏或多余的部分？顺序是否符合阅读逻辑？

### 阶段 3: 内容撰写

- **目标**: 按大纲逐章撰写文档内容
- **输入**: 阶段 2 确认的大纲
- **操作**:
  1. 按章节顺序撰写
  2. 技术内容需包含代码示例
  3. 复杂概念需配图表说明（用文字描述图表内容，标注 `[TODO: 插图]`）
  4. 保持术语一致性
- **输出**: 完整的文档初稿

### 阶段 4: 质量审查 [CHECKPOINT]

- **目标**: 检查文档质量，确保达到交付标准
- **输入**: 阶段 3 的文档初稿
- **操作**:
  1. 检查结构完整性（是否有空章节、断头路）
  2. 检查内容准确性（代码示例是否可运行、数据是否正确）
  3. 检查一致性（术语、格式、风格）
  4. 检查可读性（段落长度、专业术语解释）
  5. 生成审查报告
- **输出**: 审查报告（问题列表 + 修改建议）
- **检查要点**: 文档是否达到交付标准？审查发现的问题是否需要修改？

### 阶段 5: 修订定稿

- **目标**: 根据审查反馈修订文档
- **输入**: 阶段 4 的审查报告 + 用户反馈
- **操作**:
  1. 逐项修复审查报告中的问题
  2. 应用用户的修改意见
  3. 最终格式化（标题编号、目录生成、页眉页脚）
  4. 输出最终版本
- **输出**: 定稿文档

## 常用文档结构

**技术文档**:
1. 概述（目的、范围、前置条件）
2. 架构设计（系统架构图、组件说明）
3. 详细设计（模块设计、数据模型、接口定义）
4. 部署指南（环境要求、部署步骤、配置说明）
5. 运维指南（监控、日志、故障排查）

**API 文档**:
1. 概述（基础 URL、认证方式、通用约定）
2. 端点列表（分组、概要）
3. 端点详情（请求/响应/示例/错误码）
4. 错误处理（错误码对照表、重试策略）
5. 变更日志

**用户手册**:
1. 快速开始（5 分钟上手）
2. 功能说明（按模块/页面组织）
3. 常见问题 FAQ
4. 故障排查

## 注意事项

- 文档篇幅较短（<3页）时可以跳过阶段 2（大纲设计），直接撰写
- 如果用户提供了现有文档要求修改，阶段 1 改为"差异分析"，阶段 3 改为"增量修改"
- 代码示例应该可以实际运行，不要用伪代码糊弄
